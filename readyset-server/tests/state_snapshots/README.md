# `ControllerState` backwards-compatibility snapshots

Each `.bin` in this directory is the `rmp_serde::to_vec` encoding of a
`ControllerState` instance whose shape matches some past deployed Authority's
persisted state. The Authority writes `ControllerState` to RocksDB via that
exact same `rmp_serde::to_vec` codepath (see
`readyset-client/src/consensus/standalone.rs`), so these fixtures let us
verify a current build can still decode every shape an upgrade-in-place could
encounter.

The accompanying test `tests/state_backwards_compatibility.rs` reads each
fixture, runs it through `rmp_serde::from_slice`, and asserts the sentinel
field values registered in `readyset_server::snapshot_compat::verify_snapshot`.

## The one rule

**Snapshots are immutable artifacts of deployed state. Never regenerate an
existing snapshot.** If the wire shape changes, generate a *new* snapshot
(e.g. `v2.bin`) capturing the shape that's about to be broken, then introduce
the change.

If a snapshot test fails on your branch, you have two valid responses:

1. **Add a compat shim** (see `Config.sharding` for the recipe — keep the
   dropped field at its original position with `#[serde(default)]` +
   `#[deprecated]`, mark the containing struct `#[allow(deprecated)]`, never
   consult the field at runtime). This is the right answer when there exist
   deployed Authorities holding the now-broken state.
2. **Delete the snapshot.** Only valid when you can demonstrate no deployed
   Authority holds the relevant state. This must be justified in the commit
   message.

Regenerating an existing snapshot to "fix" the test silently re-anchors the
test to your new shape and defeats the whole mechanism. The `make_state_snapshot`
binary refuses to overwrite existing files for this reason.

## Adding a new snapshot

1. Register the name in
   `readyset-server/src/controller/snapshot_compat.rs::SNAPSHOT_NAMES`, and
   add matching `build_<name>_state` and `verify_<name>` branches.
2. Run:
   ```
   cargo run --bin make_state_snapshot -- --snapshot-name <name>
   ```
3. The tool refuses to write if `<name>.bin` already exists. It also
   self-verifies the bytes via `verify_snapshot` before writing — if
   build/verify disagree, nothing lands on disk.
4. Commit the new `<name>.bin` alongside the code changes.

## Current inventory

| Snapshot | Captures | Why we keep it |
|---|---|---|
| `v1.bin` | `ControllerState` with the legacy sharding fields populated: `Config.sharding = Some(7)`, `DfState.sharding = Some(7)`, a graph node with `Sharding::ByColumn(0, 2)`. | The sharding-removal commits on main broke decoding of state in this shape; we now carry compat shims that restore the wire shape, and this snapshot verifies they keep working. |
| `v2.bin` | `ControllerState` with the fields domain-replication removal would have broken populated to non-default values: `Config.min_workers = 5`, `Config.replication_strategy = ReaderDomains(2)`, `DfState.replication_strategy = NonBaseDomains(4)`, and a single-entry `DfState.node_restrictions`. | The domain-replication-removal work dropped the runtime use of these fields and simplified `DfState::new`'s signature accordingly; the compat shims retain the fields at their original positions and this snapshot pins the wire shape (enum variant indices, HashMap encoding, `Option<VolumeId>` framing) so any future removal that disturbs the layout fails loudly. |
| `v3.bin` | `ControllerState` with `Config.worker_request_timeout = Duration::from_secs(123)`. | Phase 3 piece A removed `Config.worker_request_timeout` from the runtime (the controller-to-worker HTTP loopback that consumed it is gone); the field stays at its original positional slot in `Config` as a `#[serde(default)] #[deprecated]` compat shim. This snapshot pins the slot specifically, so a future refactor that accidentally swaps it with the adjacent `Duration` field (`background_recovery_interval`) trips the sentinel rather than decoding silently into the wrong slot. |

### `v1.bin` provenance

The bytes were generated by serializing `ControllerState` with `rmp_serde::to_vec`
at commit `af23d30fee05`, prior to commit `bb3b60b4` ("dataflow: delete sharding
type machinery") which removed `Config.sharding`, `DfState.sharding`,
`Node.sharded_by`, and the `Sharding` enum in one move. At `af23d30fee05` those
fields and the enum still exist with their original deployed shape, so the byte
stream matches what a contemporary production Authority would have written. The
current build's `snapshot_compat::build_v1_state` produces byte-identical output,
empirically confirming the compat shims restore the exact original wire
positions and the `Sharding` variant indices are preserved.

### `v2.bin` provenance

The bytes were generated by serializing `ControllerState` with `rmp_serde::to_vec`
at commit `56550da2`, the tip of `main` immediately prior to the
domain-replication-removal stack. At that revision `Config.min_workers`,
`Config.replication_strategy`, `DfState.replication_strategy`, and
`DfState.node_restrictions` are still live native fields consumed by runtime
code, and `DfState::new` takes the latter two as constructor parameters. The
current build's `snapshot_compat::build_v2_state` produces byte-identical
output, empirically confirming the compat shims retain the original positions
for all four fields, the `ReplicationStrategy` variant indices, the single-entry
HashMap encoding for `node_restrictions`, and the `Option<VolumeId>` framing
inside `DomainPlacementRestriction`.

### `v3.bin` provenance

The bytes were generated by serializing `ControllerState` with `rmp_serde::to_vec`
at commit `4e8a2871`, the tip of `main` immediately prior to Phase 3 piece A.
At that revision `Config.worker_request_timeout` is still a live runtime field
read by the controller-to-worker HTTP RPC path; `Builder::set_worker_timeout`
and the `--worker-request-timeout-seconds` CLI flag set it. The current build's
`snapshot_compat::build_v3_state` produces byte-identical output (1239 bytes,
`cmp -l` empty), empirically confirming the compat shim restores the field at
its original positional slot (between `upquery_timeout` and
`background_recovery_interval`) and the `Duration` encoding (`{secs: u64,
nanos: u32}`) is preserved.
